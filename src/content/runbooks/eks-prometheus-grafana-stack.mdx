## Summary

This runbook deploys a production-grade Amazon EKS cluster with a fully integrated Prometheus + Grafana observability stack using the `kube-prometheus-stack` Helm chart (Prometheus Operator). Use it when standing up monitoring for a new EKS environment or rebuilding the observability layer from scratch.

End state: a highly available cluster emitting cluster, node, and pod metrics — with alerting wired to Alertmanager and dashboards served behind an ALB with TLS.

The runbook is **idempotent where possible** and includes validation gates after every major phase, making it safe to execute under change control in production.

---

## When to Use This Runbook

- Standing up a new EKS environment that requires Prometheus + Grafana from day one
- Rebuilding the observability stack after a cluster replacement or disaster recovery event
- Migrating from a self-managed Prometheus deployment to Prometheus Operator
- Reference when debugging a broken kube-prometheus-stack install

## Step 1: Set Environment Variables

Define all variables once at the top of your shell session. Every subsequent command references these — do not hardcode values inline.

```bash
export CLUSTER_NAME="prod-eks-01"
export REGION="us-west-2"
export K8S_VERSION="1.30"
export NODE_TYPE="m5.large"
export NODE_MIN=2
export NODE_MAX=4
export NODE_DESIRED=2
export GRAFANA_HOSTNAME="grafana.example.com"
export ACM_CERT_ARN="arn:aws:acm:us-west-2:123456789012:certificate/xxxxxxxx"
```

**Expected result:** `echo $CLUSTER_NAME` returns `prod-eks-01`.

---

## Step 2: Create the EKS Cluster

```bash
eksctl create cluster \
  --name "$CLUSTER_NAME" \
  --region "$REGION" \
  --version "$K8S_VERSION" \
  --nodegroup-name managed-workers \
  --node-type "$NODE_TYPE" \
  --nodes "$NODE_DESIRED" \
  --nodes-min "$NODE_MIN" \
  --nodes-max "$NODE_MAX" \
  --managed \
  --amd64 \
  --with-oidc \
  --ssh-access=false
```

**Expected result:** CloudFormation stacks (`eksctl-$CLUSTER_NAME-cluster` and `eksctl-$CLUSTER_NAME-nodegroup-managed-workers`) reach `CREATE_COMPLETE` in the AWS Console → CloudFormation.

**Validation gate:**

```bash
kubectl get nodes -o wide
# All nodes should be Ready
kubectl cluster-info
```

## Step 3: Create the Monitoring Namespace

```bash
kubectl create namespace monitoring
kubectl label namespace monitoring \
  app.kubernetes.io/managed-by=helm \
  environment=production
```

**Expected result:** `kubectl get ns monitoring` shows `Active`.

## Step 4: Install the AWS Load Balancer Controller (IRSA)

The ALB Controller uses a scoped IAM policy — **not** `ElasticLoadBalancingFullAccess`. This is intentional.

```bash
# Download the official scoped IAM policy
curl -fsSL \
  https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/main/docs/install/iam_policy.json \
  -o alb-iam-policy.json

# Create the IAM policy
aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://alb-iam-policy.json \
  --region "$REGION"

# Create the IRSA service account
eksctl create iamserviceaccount \
  --cluster "$CLUSTER_NAME" \
  --namespace kube-system \
  --name aws-load-balancer-controller \
  --attach-policy-arn "arn:aws:iam::$(aws sts get-caller-identity --query Account --output text):policy/AWSLoadBalancerControllerIAMPolicy" \
  --approve \
  --region "$REGION"

# Install via Helm
helm repo add eks https://aws.github.io/eks-charts
helm repo update

helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName="$CLUSTER_NAME" \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller \
  --set region="$REGION"
```

**Validation gate:**

```bash
kubectl get deployment -n kube-system aws-load-balancer-controller
# READY should show 2/2
kubectl logs -n kube-system \
  -l app.kubernetes.io/name=aws-load-balancer-controller \
  --tail=20
```

## Step 5: Install kube-prometheus-stack

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

helm install kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  -n monitoring \
  --set grafana.adminPassword="$(openssl rand -base64 24)" \
  --set prometheus.prometheusSpec.retention=15d \
  --set prometheus.prometheusSpec.storageSpec.volumeClaimTemplate.spec.storageClassName=gp2 \
  --set prometheus.prometheusSpec.storageSpec.volumeClaimTemplate.spec.resources.requests.storage=50Gi \
  --set alertmanager.alertmanagerSpec.replicas=2 \
  --set grafana.persistence.enabled=true \
  --set grafana.persistence.size=10Gi
```

> Store the generated Grafana password immediately — retrieve it from the Secret if lost:
> `kubectl get secret -n monitoring kube-prometheus-stack-grafana -o jsonpath="{.data.admin-password}" | base64 -d`

**Validation gate:**

```bash
kubectl get pods -n monitoring
# All pods Running or Completed; no CrashLoopBackOff
kubectl get pvc -n monitoring
# All PVCs Bound
```

## Step 6: Expose Grafana via ALB Ingress with TLS

Create `grafana-ingress.yaml`:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: grafana-ingress
  namespace: monitoring
  annotations:
    kubernetes.io/ingress.class: alb
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
    alb.ingress.kubernetes.io/certificate-arn: "<ACM_CERT_ARN>"
    alb.ingress.kubernetes.io/listen-ports: '[{"HTTPS":443}]'
    alb.ingress.kubernetes.io/ssl-redirect: "443"
spec:
  rules:
    - host: "<GRAFANA_HOSTNAME>"
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: kube-prometheus-stack-grafana
                port:
                  number: 80
```

```bash
# Substitute variables
sed -i "s|<ACM_CERT_ARN>|$ACM_CERT_ARN|g; s|<GRAFANA_HOSTNAME>|$GRAFANA_HOSTNAME|g" grafana-ingress.yaml
kubectl apply -f grafana-ingress.yaml
```

**Validation gate:**

```bash
kubectl get ingress -n monitoring grafana-ingress
# ADDRESS column should show an ALB DNS name (may take 2-3 minutes)
```

Create a Route 53 ALIAS record pointing `$GRAFANA_HOSTNAME` to the ALB DNS name from above.

**Console path:** Route 53 → Hosted Zones → [your zone] → Create Record → ALIAS → Application and Classic Load Balancer

## Rollback

```bash
# Remove observability stack
helm uninstall kube-prometheus-stack -n monitoring
kubectl delete namespace monitoring

# Remove ALB Controller
helm uninstall aws-load-balancer-controller -n kube-system
eksctl delete iamserviceaccount \
  --cluster "$CLUSTER_NAME" \
  --name aws-load-balancer-controller \
  --namespace kube-system \
  --region "$REGION"

# Delete cluster (destructive — confirm before running)
eksctl delete cluster --name "$CLUSTER_NAME" --region "$REGION"
```

## Cost Considerations

Running this stack 24/7 in `us-west-2` with 2× m5.large nodes incurs approximately:

| Resource | Estimated Monthly Cost |
|---|---|
| 2× m5.large EC2 (On-Demand) | ~$140 |
| EBS gp2 60 GB total | ~$6 |
| NAT Gateway (single AZ) | ~$35+ |
| ALB | ~$20 |
| **Total (approx)** | **~$200+** |

Use Savings Plans or Spot for non-production. Enable VPC endpoints for ECR, STS, and S3 to reduce NAT costs in production.

## Escalation

| Condition | Next Action |
|---|---|
| ALB Controller pods in CrashLoopBackOff | Check IRSA annotation on service account; verify OIDC provider is registered |
| PVCs stuck Pending | Verify EBS CSI driver is installed (`kubectl get pods -n kube-system -l app=ebs-csi-controller`) |
| Grafana Ingress ADDRESS blank after 5 min | Check ALB Controller logs for annotation errors |
| eksctl cluster creation fails at CFN | Check CloudFormation stack events in Console for the root cause |
